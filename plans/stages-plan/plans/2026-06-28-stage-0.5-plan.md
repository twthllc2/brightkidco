**Owner:** Ayoub | **Date:** June 28, 2026 | **Status:** Plan ready

> **Purpose:** Build a per-level journey that prevents desensitization, preserves trust at each emotional phase, and gives Stage 1 subagents a CONCRETE blueprint for each email — not a useless table.

## 0. WHAT THIS STAGE ACTUALLY PRODUCES

For each level (L1, L2, L3, GF), a **Journey Document** that contains:

1. **The sequence** — every email the customer receives, in order, with timing
2. **The emotional arc** — what the customer is FEELING at each touchpoint
3. **The structural blueprint per email** — 9 dimensions filled in, justified
4. **The variance ledger** — running tally of patterns used, what's still available
5. **The trust meter** — predicted trust level before/after each email
6. **The conversion gates** — which emails can prompt a purchase, which cannot
7. **The dead-ends** — what happens if customer ignores each email
8. **Cross-flow continuity** — how Welcome E3 should feel different from Cart E2 even if same customer gets both

This is NOT a 4-row table. It's a 30+ page document per level.

## 2. THE ULTRA-DETAILED STRUCTURE PER LEVEL

Each Journey Document (one per level) has these sections:

### Section A — AVATAR PROFILE (full context)
- Who this mom is (age, situation, child specifics)
- What she's already tried (failed methods)
- Where she is emotionally when she enters the program
- Her trust threshold (what destroys it)
- Her decision speed (2-7 days / 7-21 days / 14-30+ days)
- Her hope/realism starting ratio
- What she needs from us

### Section B — THE SEQUENCE (chronological email list)
For each email in order:
- Email ID + position
- Day/hour sent
- Trigger (what caused this email to fire)
- Goal (what we want her to DO after reading)

### Section C — THE EMOTIONAL ARC (per email)
For each email:
- **Before:** what she's feeling right before opening
- **Hook moment:** first sentence — does she recognize herself?
- **Trust transaction:** what does she need to believe for this email to land?
- **After:** what she's feeling after closing
- **Trust delta:** -2 / -1 / 0 / +1 / +2

### Section D — THE STRUCTURAL BLUEPRINT (per email)
For each email, all 9 dimensions filled:
1. Section count + section names (from Stage 0 extraction)
2. Section pattern (A-H from email-marketing-structure skill)
3. Image density target (1-6)
4. Color temperature (warm/cool/neutral/dark)
5. Email length (short/medium/long + word count estimate)
6. Argument structure (PAS/AIDA/Story/Data/Testimonial/Direct)
7. Visual rhythm (tight/open/mixed)
8. CTA position (early/mid/late/double-tap/text-link)
9. Subject line type (observation/question/mechanism/story/validation/data/quote)

PLUS for each: WHY this choice, not just WHAT.

### Section E — THE VARIANCE LEDGER (running)
- Pattern used by email N-3, N-2, N-1
- Pattern available for email N
- Constraint: must differ from at least 6 of 9 dimensions vs N-1 and N-2
- If constraint can't be met (too many emails in same flow with similar needs), flag for special handling

### Section F — THE TRUST METER (per email)
Predicted trust level before/after each email:
- Email 1: trust X → trust Y (delta)
- Email 2: trust Y → trust Z (delta)
- ...
- Email N: trust ? → trust ?

If trust ever drops below threshold (different per level), flag for intervention.

### Section G — CONVERSION GATES (per email)
Which emails can prompt purchase:
- Hard CTAs allowed (L1)
- Soft CTAs allowed (L2)
- No CTAs / text links only (L3)
- Permission-not-to-buy required (L3)

### Section H — DEAD-ENDS (per email)
What happens if customer ignores this email:
- Does winback fire?
- Does sunset start?
- Does next email skip?
- How many ignores before segment changes?

### Section I — CROSS-FLOW CONTINUITY
How this email should feel vs the previous email a customer got:
- If customer got Welcome E2 yesterday and Cart E1 today, they shouldn't feel like they got two of the same email
- Cross-flow variance rules:
  - Different color temperature OR different image density OR different argument structure
  - Not the same subject line type twice in 48 hours

### Section J — FAILURE MODES
What if the email is mis-built:
- Wrong level calibration → trust crash
- Forbidden phrase → unsubscribe
- CTA in wrong position → wrong action taken
- Visual rhythm mismatch → fatigue

## 3. SUBAGENT JOBS

### Job Type 1 — Avatar Deep-Dive (4 jobs, one per level)
For each level (L1, L2, L3, GF):
- Read the synthesis docs for that level
- Read the demands doc sections for that level
- Produce Section A (Avatar Profile) as a standalone document

Output: `/root/projects/brightkidco/_notes/journey-l{l}-01-avatar.md`

### Job Type 2 — Sequence Builder (4 jobs, one per level)
For each level:
- Read locked-profile.json for all emails that apply to this level
- Build Section B (The Sequence) — every email this level receives in chronological order
- Include trigger, day/hour, goal per email

Output: `/root/projects/brightkidco/_notes/journey-l{l}-02-sequence.md`

### Job Type 3 — Emotional Arc + Trust Meter (4 jobs)
For each level:
- For each email in the sequence, build the BEFORE/HOOK/TRUST-AFTER/DELTA narrative
- Build the running trust meter
- Flag any email where trust delta is negative or where trust drops below threshold

Output: `/root/projects/brightkidco/_notes/journey-l{l}-03-emotional-trust.md`

### Job Type 4 — Structural Blueprint + Variance Ledger (4 jobs)
For each level:
- For each email, fill all 9 dimensions + WHY
- Build running variance ledger
- Flag any email where 6/9 variance can't be achieved

Output: `/root/projects/brightkidco/_notes/journey-l{l}-04-structural-blueprint.md`

### Job Type 5 — Conversion Gates + Dead-Ends + Cross-Flow (4 jobs)
For each level:
- Per email: conversion gate, dead-end behavior, cross-flow continuity rules

Output: `/root/projects/brightkidco/_notes/journey-l{l}-05-flow-behavior.md`

### Job Type 6 — Journey Assembler (4 jobs, one per level)
For each level:
- Combine sections A-J into one Journey Document
- Cross-check for internal consistency
- Output the master document

Output: `/root/projects/brightkidco/_notes/journey-l{l}-MASTER.md`

### Job Type 7 — Cross-Level Continuity Checker (1 job)
- Compare all 4 master journeys
- Identify cross-level inconsistencies (e.g., L1 and L2 both planned for same color in same week)
- Output cross-level sync report

Output: `/root/projects/brightkidco/_notes/journey-cross-level-sync.md`
