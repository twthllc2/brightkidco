# STAGE 0 — PRECHECK PLAN (v3 — no scripts, subagents only)

**Owner:** Ayoub | **Date:** June 28, 2026 | **Status:** Ready to execute

> **Core rule:** NO SCRIPTS for design decisions. Scripts are blind. Subagents see and reason.
>
> **Stage 0 reconciles 125 copy files against the strategy macro AND characterizes each file's internal body structure BEFORE any wireframe work begins. Surface divergences and inconsistencies in both metadata AND copy body shape so we resolve them once, not 125 times.**

---

## 0. WHY STAGE 0 EXISTS

Two prior versions were wrong:

- **v1** only reconciled metadata headers. Missed that body structure varies wildly across 125 files.
- **v2** added body extraction BUT proposed 4 Python scripts for it. Scripts are blind — they can't distinguish a real section header from a bold emphasis paragraph, can't decide which of two overlapping shapes a file matches, can't tell when "missing" data is intentional.

**v3 rule:** Every decision that requires SEEING the file goes to a subagent. Scripts only do pure file I/O and counting (after a subagent has decided what to count).

## 3. PHASE 1 — CHARACTERIZATION (501 subagents)

### 3.1 Subagent Type A — Shape Detector (125 jobs)

**Input per job:**
- ONE copy file path
- The 5 known shape definitions (with sample files)
- Instruction: read the file, decide shape, OR propose new shape

**Process:**
1. Open the file
2. READ the entire file (don't skim)
3. LOOK for:
   - Where is the email body located?
   - How are sections delimited within the body?
   - What markers separate one section from another?
4. MATCH against the 5 known shapes
5. If matches: output the shape name
6. If NO match: 
   - Describe what you see (where body is, how sections are marked)
   - Propose a NEW shape (give it a name like "Shape F" and describe the detection rule)
   - Provide 1-2 example files that match this shape

**Output per job:**
```json
{
  "file_path": "outputs/copy/welcome/welcome-01-e1-l1.md",
  "shape_matched": "Shape A (welcome-style)",
  "confidence": "high",
  "reasoning": "File has '## Email Body' header followed by '### Section 1: Hook', '### Section 2: Validation', etc. Matches Shape A definition exactly."
}
```

OR for new shapes:
```json
{
  "file_path": "outputs/copy/[some-flow]/[some-file].md",
  "shape_matched": "NEW: Shape F",
  "confidence": "high",
  "reasoning": "File uses ## Section A: [Name], ## Section B: [Name] pattern with no preceding ## Email Body header. Different from Shape A in that section markers are at ## level, not ### level.",
  "proposed_shape_definition": {
    "name": "Shape F (flat-section)",
    "body_start_marker": "^## Section A:",
    "body_end_marker": "^## [A-Za-z]+ Sign-Off|^## Footer",
    "section_marker_pattern": "^## Section [A-Z]: (.+)$",
    "example_files": ["[file1].md", "[file2].md"]
  }
}
```

**Output file:** /root/projects/brightkidco/_notes/detected-shapes.json (one entry per file)

### 3.2 Subagent Type B — Metadata Extractor (125 jobs)

**Input per job:**
- ONE copy file path
- The 3 known metadata patterns (table / blockquote+table / embedded)
- Instruction: read the file, find the metadata, extract fields

**Process:**
1. Open the file
2. SCAN the first 3000 chars for metadata
3. Look for:
   - Pattern A: `## Metadata` table → parse rows
   - Pattern B: blockquote at top with `**Flow:**`, etc.
   - Pattern C: H1 title + blockquote with embedded fields
4. EXTRACT each field:
   - Flow (e.g., "Welcome Flow", "Cart Abandonment (Flow 02)")
   - Position (e.g., "E1", "E2, 24 Hours", "D14")
   - Level (e.g., "L1, Sarah", "Cross-Level", "GF Only")
   - Phase (e.g., "Phase 1", "Phase 2", "Phase 3")
   - Purpose (the "what this email does" line)
   - Section structure (when declared)
   - Framework (when declared, e.g., "PAS", "AIDA")
   - Sign-off variant (when declared)
5. If a field is NOT present in the file: mark as `MISSING` (don't guess)
6. If the metadata uses unusual wording: capture the raw text + a normalized version

### 3.5 Phase 1 orchestration

Order:
1. Run Shape Detector (125 subagents) FIRST — its output feeds the Section Extractor
2. Run Metadata Extractor (125 subagents) in parallel — independent
3. Run Section Extractor (125 subagents) AFTER Shape Detector completes
4. Run Strategy Parser (1 subagent) in parallel with everything else

Batch sizing: 6 concurrent subagents (avoid system overload)
- Shape Detector: 125/6 = 21 batches × ~5 min = ~1.75 hours
- Metadata Extractor: 125/6 = 21 batches × ~3 min = ~1 hour (faster, simpler task)
- Section Extractor: 125/6 = 21 batches × ~7 min = ~2.5 hours (slowest, most reasoning)
- Strategy Parser: 1 batch × ~15 min

## 4. PHASE 2 — RECONCILIATION (126 subagents)

### 4.1 Subagent Type E — Reconciler (125 jobs)

**Input per job:**
- ONE email ID (e.g., "welcome-01-e1-l1")

- Extracted sections for that email (from Subagent Type C)
- Detected shape (from Subagent Type A)
- Strategy macro for the relevant flow (from Subagent Type D)
- Demands doc sections relevant to this email's flow
- The precedence rule: Demands > Synthesis > Copy metadata > Strategy macro

**Process:**
1. Read all inputs
2. CHECK each of the 9 reconciliation dimensions:

**Dim 1 — Flow Identity:**
- Copy metadata's flow vs strategy's flow vs filename folder
- If all 3 match: ✅
- If file is in `welcome/` but declares "Cart" → FILE MISPLACED, flag

**Dim 2 — Position Within Flow:**
- Copy's position vs strategy's position
- Welcome E1 must be Day 0
- Cart E2 must be 24 hours
- etc.

**Dim 3 — Level Assignment:**
- Copy's level vs strategy's level vs filename
- Welcome files: should be l1/l2/l3/gf
- Cart files: should be cross-level
- PP-Level Detection: should be gf only
- Check filename suffix matches declared level

**Dim 4 — Phase Assignment:**
- Copy's phase vs strategy's phase
- Already known conflict: gap #6 in GIULIANO-DEMANDS §20

**Dim 5 — Purpose / Lever Alignment:**
- Copy's declared purpose vs strategy's copy_strategy
- Decide if intent aligns (different wording, same goal = OK)

**Dim 6 — Section Count + Names:**
- Copy's declared section structure (if any)
- Body extraction's actual section count + names
- Strategy's implied section structure
- All 3 should match
- If declared says "4 sections" but extraction found 3 → EXTRACTION might be wrong OR copy is inconsistent

**Dim 7 — Sign-Off Variant:**
- Copy's declared variant (if any) vs strategy's inferred variant
- Welcome E1-E2 → SHORT
- Welcome E3-E5 → MEDIUM
- Welcome E6+ and winback → LONG

**Dim 8 — Body Shape Match (NEW):**
- Detected shape is recognized (Shape A/B/C/D/E)
- If UNKNOWN/new shape: flag for Ayoub review

**Dim 9 — Body Content Quality (NEW):**
- Word count reasonable for position? (E1 = short, E7 = long)
- Section word distribution makes sense? (no single section 90% of body)
- Lena voice present in at least 1 section?
- If body is missing or empty: CRITICAL flag

3. CLASSIFY each divergence:
   - **Class A** — auto-fixable (naming, formatting)
   - **Class B** — precedence-resolvable (apply precedence rule)
   - **Class C** — Ayoub decision needed (write the question)

4. For Class C items, write the EXACT question for Ayoub:
   - What's the conflict
   - What each source says (with file:line citations)
   - What the impact is
   - What you recommend (default action)

## 8. RISKS

### Risk 1 — Subagent drift
125 subagents might use slightly different criteria for "what counts as a section" or "what level is this."

**Mitigation:** TEMPLATE prompts (above). Every subagent of a given type gets the SAME prompt structure with only the file path swapped. Ayoub approves templates before launch.

### Risk 2 — New shapes emerge late
If 50 files are processed and Shape F is discovered, the first 50 might need re-classification.

**Mitigation:** run Shape Detector in TWO PASSES:
- Pass 1: process 10 sample files (1-2 per folder), discover all shapes
- Ayoub approves new shape definitions
- Pass 2: process all 125 files with the complete shape library

### Risk 3 — Reconciler can't decide without seeing more context
Some Class C items need cross-email reasoning (e.g., "this email's section count = 5 but all other Welcome E1 variants have 4").

**Mitigation:** Reconciler subagent has access to ALL extracted data for the flow, not just one email. It can compare across level variants.

### Risk 4 — 502 subagent jobs is expensive
At ~5 min each, ~7 hours compute. Long.

**Mitigation:** 
- Run subagents in batches of 6-10 concurrent
- Run across multiple sessions (start in one session, continue in next)
- Stage 0 does NOT need to complete before Stage 1 starts on the FIRST FLOW. Once Welcome flow's 32 files are characterized, Stage 1 can start on those 32 wireframes.

## 9. EXECUTION ORDER

```
SESSION 1 (or until paused):
  Pass 1: Shape Detector on 10 sample files → discover shapes
  Ayoub approves new shape definitions
  Pass 2: Shape Detector on all 125 files (in parallel batches of 6)

SESSION 2 (or continue):
  Metadata Extractor on all 125 files (parallel)
  Strategy Parser (1 job)

SESSION 3 (or continue):

  → Class C questions collected

SESSION 4 (Ayoub-driven):
  Ayoub reviews Class C questions, makes decisions
  Locked Profile Builder assembles final JSON

SESSION 4+:
  Stage 0 DONE
  Stage 0.5 (customer journey matrix) can begin
  Stage 1 (wireframes) can begin on characterized flows
```

## 10. WHAT STAGE 0 DOES NOT DO

- Stage 0 does NOT build wireframes.
- Stage 0 does NOT pick components.
- Stage 0 does NOT build HTML.
- Stage 0 does NOT make design decisions about visual treatment.

Stage 0 is purely: READ → LOCATE BODY → EXTRACT SECTIONS → COMPARE → DOCUMENT → DECIDE → LOCK.
