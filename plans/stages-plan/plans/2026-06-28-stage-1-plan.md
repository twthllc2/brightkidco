# STAGE 1 ASCII WIREFRAME PLAN — REVISED

**Owner:** Ayoub | **Date:** June 28, 2026 | **Status:** Locked

> **THE RULE (non-negotiable):** Every section follows this stack:
> HEADER (logo, static) → HEADLINE + SUBHEADLINE → IMAGE or COMPONENT → CTA → NEXT SECTION
>
> NO side-by-side image/text layouts. NO image-LEFT, text-RIGHT. NO two-column. NEVER.
>
> Stack vertically. Always. That's how email works on mobile and desktop.

---

## 1. WHAT A WIREFRAME IS

A wireframe is the visual blueprint of an email — drawn as ASCII — that:

- Defines the ROLE of each visual element (NOT the specific component)
- Stages 1's deliverable to Stage 2
- Stage 2 PICKS the actual components and images from the libraries

**Stage 1 = intent. Stage 2 = selection.**

---

## 2. THE UNIVERSAL SECTION STACK (every section follows this)

```
┌──────────────────────────────────────────────────────────────┐
│  [ROLE: headline + subheadline — text only, no images]        │
│                                                              │
│  [ROLE: image OR component — picked from libraries]            │
│                                                              │
│  [ROLE: cta — if section is CTA-bearing, otherwise skip]       │
└──────────────────────────────────────────────────────────────┘
                              ↓ gap (varies 40-56px) ↓
```

Every section. No exceptions. The HEADLINE comes FIRST, then IMAGE/COMPONENT, then CTA.

## 4. WHAT STAGE 1 SUBAGENT MUST PRODUCE

For each of 125 emails, the subagent produces ONE wireframe file (appended to the copy file) containing:

### 4.1 Subject + Preheader line
```
SUBJECT: [exact subject from copy file]
PREHEADER: [exact preheader from copy file, must complement subject — not repeat]
```

### 4.2 Section-by-section ASCII blocks

Each section = ONE ASCII box with this structure:

```
[NAME: section-N-role]
─────────────────────────────────────────────────────────────────
HEADLINE: [exact headline text from copy file section]
SUBHEADLINE: [exact subheadline text if section has one, else "none"]

[VISUAL ELEMENT]
ROLE: [what this element does, NOT what component it is]
INTENT: [what it should communicate visually]
EXAMPLE_PICK: [Stage 2 will pick actual — e.g., "lifestyle photo of kid 3-5, hopeful, real home"]

[CTA if applicable]
BUTTON_TEXT: [exact button text from copy file]
BUTTON_INTENT: [what click does — e.g., "go to PDP"]
─────────────────────────────────────────────────────────────────
```

### 4.3 Below-fold check
After listing all sections, the subagent must answer:
```
ABOVE_FOLD_CHECK:
- Section 1 visible above 400px (mobile)? YES/NO
- Section 1 headline visible above 400px? YES/NO
- If NO, section 1 is too long — adjust
```

### 4.4 Image density count
```
IMAGE_COUNT:
- Total images in this email: N
- Image positions: [list which sections have images]
- Density category: 1 (text-only) / 2-3 (balanced) / 4-5 (image-heavy) / 6 (gallery)
```

### 4.5 Component intent count
```
COMPONENT_COUNT:
- Total distinct component ROLES in this email: N
- Roles: [list them]
- Density category: 3-4 (typical) / 5-6 (rich) / 7-8 (very rich)
```

### 4.6 Structural intent (9 dimensions from variance tracker)
```
STRUCTURAL_INTENT:
- Section count: N
- Section pattern (A-H): [which one]
- Image density: 1-6
- Color temperature: warm/cool/neutral/dark (state it — subagent decides based on macro intent)
- Email length: short/medium/long
- Argument structure: PAS/AIDA/Story/Data/Testimonial/Direct
- Visual rhythm: tight/open/mixed
- CTA position: early/mid/late/double-tap/text-link
- Subject line type: observation/question/mechanism/story/validation/data/quote
```

### 4.7 Variance check
```
VARIANCE_CHECK:
- Previous email in flow+level: [email_id]
- Differences across 9 dimensions: N/9
- If <5: ADJUST before saving (change one dimension, re-check)
```

## 5. WHAT STAGE 2 PICKS (the selection job)

Stage 2 reads the wireframe and PICKS:

### 5.1 From 36 product images
For each `EXAMPLE_PICK: lifestyle photo of kid 3-5` line, Stage 2:
1. Opens `/root/projects/brightkidco/outputs/email-assets/product-photos/lifestyle/`
2. Sees 11 images
3. Picks the one that matches intent
4. Checks image-usage tracker (cap 25x)
5. Assigns the filename to the ROLE

### 5.2 From 949 component catalog
For each `ROLE: callout-quote-highlight` line, Stage 2:
1. Opens `/root/projects/brightkidco/outputs/email-assets/component-catalog.json`
2. Filters by `position: body`, `messaging: proof`, `audience: L1`
3. Browses the candidates visually (or via subagent)
4. Picks the one that matches intent (cream card vs stat box vs quote block)
5. Checks component-usage tracker (cap 8x)
6. Assigns the component ID to the ROLE

### 5.3 Color system
Applies System B (already locked) automatically:
- Brand-Green #039902 (CTAs, accents)
- Mint #DBFFCD (highlights)
- Cream Card #FBF7EB (cards)
- Yellow #F5C84B (L1 stats only)
- Dark Text #1A1A1A (body)
- Body Gray #4A4A4A (captions)
- Off-White #FAF9F7 (background)
- Soft Gray #9AA8A7 (subtle text)

Stage 1 wireframe does NOT specify colors. Stage 2 applies them automatically based on the role.

---

## 6. THE WIREFRAME IS NOT THE COMPONENT

The wireframe says:
- ROLE: callout-quote-highlight

Stage 2 turns this into:
- COMPONENT ID: `CALLOUT-CREAM-3` (or whatever exists in 949 catalog)
- HTML: actual <table><tr><td>...</td></tr></table> markup

Stage 1 NEVER names a specific component. NEVER says "Cream Card". NEVER says "Lena sign-off block v2".

Stage 1 says ROLE. Stage 2 picks COMPONENT.

## 9. WHAT STAGE 1 NEVER DOES (forbidden in wireframes)

❌ NEVER name a specific component ID (e.g., `CALLOUT-CREAM-3`)
❌ NEVER name a specific image file (e.g., `toddler-happy-home.jpg`)
❌ NEVER specify a specific hex color (System B is automatic)
❌ NEVER specify image-text side-by-side layout (always stacked)
❌ NEVER specify two-column layouts (always single column)
❌ NEVER use image-LEFT text-RIGHT pattern
❌ NEVER put CTA before the mechanism/validation
❌ NEVER recommend a specific size for fonts (Stage 2 picks from system)

## 11. SUBAGENT PROMPT TEMPLATE

```
You are building an ASCII WIREFRAME for BKC email {email_id}.

INPUTS:
1. /root/projects/brightkidco/outputs/copy/{flow}/{filename}.md (the copy)
2. /root/projects/brightkidco/_notes/locked-profile.json (entry for this email)
3. /root/projects/brightkidco/_notes/journey-{level}-MASTER.md (if level-specific)
   OR /root/projects/brightkidco/_notes/journey-gf-MASTER.md (if cross-level)
4. /root/projects/brightkidco/_notes/journey-cross-level-sync.md (for conflicts)

PROCESS:
1. Read the copy file. Identify every section (Hook/Validation/Mechanism/CTA, etc.)
2. For each section, identify:
   - The headline text (if any)
   - The subheadline text (if any)
   - The body text
   - Whether this section has an image OR component OR both OR neither
   - Whether this section has a CTA
3. Build the wireframe:
   - SUBJECT + PREHEADER from copy file
   - Header (static) — note role only
   - For each section, produce an ASCII block with:
     * HEADLINE: (verbatim from copy)
     * SUBHEADLINE: (if any)
     * [VISUAL ELEMENT] section with ROLE + INTENT + EXAMPLE_PICK
     * [CTA] section (if applicable)
   - Lena sign-off block (SHORT/MEDIUM/LONG per position)
   - Footer (static)
4. Stack vertically — NEVER side-by-side image/text
5. Above-fold check
6. Image count + component count
7. 9-dimension structural intent
8. Variance check vs previous email in flow+level

CONSTRAINTS (NON-NEGOTIABLE):
- NEVER name a specific component ID (use ROLE names)
- NEVER name a specific image file (use ROLE + INTENT)
- NEVER specify hex colors (System B is automatic)
- NEVER specify image-text side-by-side layout
- NEVER use two-column layouts
- ALWAYS: header → headline → image/component → CTA
- ALWAYS: image/component goes BELOW headline, not beside

OUTPUT:
- Wireframe appended to the bottom of the copy file
- Start with "## WIREFRAME — built {date} by subagent-N"
- Follow the format spec in the Stage 1 plan

REPORT BACK: "Email {email_id} wireframed. Sections: N. Components: M. Images: I. Variance: N/9. Status: ✅."
```
