Previous Stage 2 attempts ignored two of Ayoub's core rules:
1. **4-6 images per email** — was not enforced. 62 emails ended up with 0 images, 18 with 1, ~20 with 2-3. Only 18 had 4+ images.

This version enforces BOTH constraints. Each subagent picks 4-6 images and a unique hero for each email.

## 3. THE HARD RULES (enforced in every subagent prompt)

### 3.1 Component count
- **Minimum 4 components** per email (NOT 3, NOT 2)
- **Maximum 8 components** per email (the 8-cap)
- If the wireframe has fewer ROLE slots than 4, the subagent MUST add additional components (e.g., a testimonial card, a guarantee block, a "next steps" navigation component) until the count reaches 4

### 3.2 Image count
- **Minimum 4 images** per email
- **Maximum 6 images** per email
- If the wireframe doesn't specify 4+ images, the subagent MUST add images to body sections
- Image picks come from the 36-image library, capped at 25 uses per image
- L3 emails may use FEWER IMAGES (text-heavy is OK) but minimum is 4 — find 4 appropriate L3-toned images (cream, earth tones, calm)

### 3.3 Hero component
- **NEVER reused** across emails. Each email gets a UNIQUE hero component.
- The 8-cap does NOT apply to hero. Hero is hard-capped at 1 use (this email only).
- If the tracker says a hero component was used in another email, SKIP it and pick a different one.
- Exception: product_showcase and image_showcase CAN be reused (per Ayoub's brief).

### 3.4 Body sections
- Minimize reuse — ideally each section pick is unique to this email
- The 8-cap is the HARD LIMIT
- If a component is at 8, skip it

### 3.5 Other rules (from previous plan, still valid)
- Component cap 8 (HARD STOP at 8, no exceptions)
- Image cap 25
- No teal #2BAEB4 (System A Giuliano, banned)
- No scripts for design decisions
- One subagent = one section
- L3: zero yellow, cream/earth only, text-link CTAs
- GF: R1-R6 strict, NO autism/ASD/Level labels
- L1: bright greens + yellow stat accents
- L2: muted greens + BCBA acknowledgment

## 4. JOB STRUCTURE (per email)

For each of 125 emails, Stage 2 dispatches `N` subagents where N = number of distinct ROLE sections in the wireframe (typically 3-8 per email).

**For an email with 5 sections: 5 subagents. For an email with 8 sections: 8 subagents.**

Plus 1 image-augment subagent per email that runs AFTER the section picks and adds any missing images to reach the 4-image minimum.

Plus 1 cap-validation subagent per email that checks the final picks respect all caps.

**Total Stage 2 subagent jobs: ~625 section picks + 125 image-augment + 125 cap-validation = ~875 subagent jobs.**

## 5. SUBAGENT PROMPT (per section, with image-cap enforcement)

Every section subagent gets THIS prompt (with only email_id, flow, section_index, and role_slot changing):

```
You are picking the right asset for ONE section of ONE BKC email.

CRITICAL RULES (from Ayoub's 28 June brief):
- Email needs 4-6 IMAGES total. This section's image pick counts toward that minimum.
- Email needs 4-8 COMPONENTS total. This section's component pick counts toward that minimum.
- HERO components are NEVER reused across emails (each email gets unique hero).
- Other components: minimize reuse, cap at 8.
- ONLY product_showcase + image_showcase can be reused.

INPUTS:
1. Copy file: /root/projects/brightkidco/outputs/copy/{flow}/{filename}.md
   - Read the WIREFRAME (lines from "## WIREFRAME — built" to "END OF WIREFRAME")
   - Find the ROLE for section {N} (the section you're picking for)
2. Locked profile: /root/projects/brightkidco/_notes/locked-profile.json — find your email_id
3. Journey master: /root/projects/brightkidco/_notes/journey-{level}-MASTER.md — read the section about visual calibration for your level
4. Component catalog: /root/projects/brightkidco/outputs/email-assets/component-catalog.json — 949 components, filter by position matching the ROLE and audience matching the level
5. Component showcase: /root/projects/brightkidco/outputs/email-design-environment/_extracted/showcase-artifacts-final.html — visual reference
6. Image library: /root/projects/brightkidco/outputs/email-assets/product-photos/ — 36 images across 5 subfolders
7. Component-usage tracker: /root/projects/brightkidco/_notes/component-usage.json — read counts, skip at 8
8. Image-usage tracker: /root/projects/brightkidco/_notes/image-usage.json — read counts, skip at 25

TASK:
1. Read the wireframe's ROLE slot for section {N}
2. Read the INTENT — what this section should communicate
3. Read the level-specific calibration rules
4. Browse the component catalog: filter by ROLE-matching position and audience
5. Open the showcase HTML to visually inspect top 3-5 candidates
6. **IF ROLE = hero**: SKIP any hero component already used in ANY other email (zero reuse)
7. **IF ROLE = body-section**: pick a component with low usage (prefer 0-3 uses across all emails)
8. **IF ROLE = product_showcase OR image_showcase**: these CAN be reused (per Ayoub's brief)
9. Skip any component already at 8 uses
10. Skip any component already used in this same email (no duplicates)
11. Skip any teal-colored component (#2BAEB4)
12. Skip components with level-incompatible audience (e.g., L2-only for L3)
13. PICK the best-fit component. Document WHY.
14. **PICK AN IMAGE** (if the section needs one). For body sections, default to picking an image (this email needs 4-6 total).
15. Skip images at 25 uses
16. Skip images used in this same email
17. Image must match INTENT and level calibration
18. Document alt text for the image per V23 level spec
19. APPEND the structured record to the copy file under "## STAGE 2 PICKS — {email_id}"
20. UPDATE component-usage.json and image-usage.json with your picks

HERO COMPONENT RULE (CRITICAL):
- If you're picking the HERO section, the component MUST be unique to this email
- Search the entire tracker for any hero component already used at count > 0
- Skip all of those — pick a fresh hero component
- The exception is product_showcase and image_showcase, which CAN be reused

IMAGE COUNT RULE (CRITICAL):
- After your pick, the email's total image count must move toward 4-6
- If this is the only image pick so far and the email is 0-image so far, you MUST pick an image
- If the email already has 4+ images, this section can be image-less
- The 36-image library has enough variety to always have an image for any section

VARIANCE CHECK:
- If this is NOT the first section of this email, check what was picked for the previous section
- Avoid picking the SAME component ID for two adjacent sections in the same email
- If variance is at risk, prefer a different component even if slightly less optimal

CONSTRAINTS:
- Component cap: 8 uses across all 125 emails (HARD STOP at 8)
- Hero components: NEVER reused (effectively cap 1, but prefer unique)
- Image cap: 25 uses across all 125 emails
- Level rules per journey-MASTER (L3 no yellow, etc.)
- Do NOT use teal #2BAEB4 (System A — explicitly banned)
- Do NOT introduce HTML, CSS, or any email markup
- Do NOT change the copy text

REPORT BACK:
"Email {email_id} section {N} picked. Component: {ID} (now N/8 used). Image: {filename} (now N/25 used). Email image total: N/6. Status: ✅."
```

## 6. IMAGE-AUGMENT SUBAGENT (one per email, runs after section picks)

For each email, if the section picks only added 0-3 images, dispatch 1 more subagent to add the missing images.

```
You are augmenting image picks for BKC email {email_id}.

CURRENT STATE:
- Section picks completed for this email
- Total images in the email so far: N (must reach 4 minimum)

TASK:
1. Read the current STAGE 2 PICKS section in the copy file
2. Identify which body sections DO NOT have an image yet
3. For 4-6 such sections, PICK an image from the 36-image library that:
   - Matches the section's INTENT
   - Matches the level calibration (L1 bright, L2 muted, L3 earth, GF universal)
   - Has not been used in this email yet
   - Has current usage < 25 in image-usage.json
4. APPEND new image picks to the STAGE 2 PICKS section as "## Additional images" subsection
5. UPDATE image-usage.json
6. ENSURE final image count is 4-6 (if over 6, remove the most redundant ones)

REPORT:
"Email {email_id} image augmentation complete. Final image count: N/6. Status: ✅."
```

---

## 7. CAP-VALIDATION SUBAGENT (one per email, runs last)

For each email, after all section picks + image augmentation, validate:

```
You are validating caps for BKC email {email_id}.

CHECK:
1. Read the STAGE 2 PICKS section
2. Count: how many images total? Must be 4-6
3. Count: how many components total? Must be 4-8
4. Read the component-usage.json — does the email's hero component have a count > 1? (If yes, REPLACE with a fresh hero)
5. Verify all components respect the 8-cap
6. Verify all images respect the 25-cap
7. Verify no teal components
8. Verify level calibration

REPORT:
"Email {email_id} validation: components N/8 ✅, images N/6 ✅, hero unique ✅, no teal ✅. Status: ✅."
```
